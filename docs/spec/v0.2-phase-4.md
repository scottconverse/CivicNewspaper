## 4a. Source Tier classification

**Modify `src-tauri/migrations/0004_source_tier.sql`** (UTF-8 LF):

```sql
ALTER TABLE sources ADD COLUMN tier TEXT NOT NULL DEFAULT 'official_record'
  CHECK (tier IN ('official_record', 'news_reporting', 'community_signal'));

-- Backfill: any existing source with type='media_lead' becomes news_reporting.
UPDATE sources SET tier = 'news_reporting' WHERE type = 'media_lead';

-- All other existing sources default to official_record per the migration default.
```

The taxonomy is from `src-tauri/prompts/source-tier-reference-v4.md` (imported in Pre-Phase 0). It is **deliberately not the strict v2.3 Tier-A/B/C/D system** — `news_reporting` is a legitimate primary source for publication, not a "leads only" downgrade.

**Modify `src-tauri/src/core/db.rs`** — `Source` struct gains a `tier` field. CRUD updated to accept/return it.

**Modify `src/components/Sources*.tsx`** (or wherever sources are managed) — add a tier selector when adding/editing a source. Render the source's tier as a badge in the LeadQueue and PublishPanel.

## 4b. Bundled prompt library access

**Create `src-tauri/src/core/prompts.rs`** with:

- `load_prompt(name: &str) -> Result<String, String>` — reads from the bundled `prompts/` resource directory using `tauri::api::path::resolve_resource`. Returns the raw prompt text.
- Tauri command `list_prompts() -> Vec<PromptMeta>` — returns `[{ id, category, title, path, description }]` enumerated from the bundled directory. Categories: `aggregator`, `story`, `audit`, `utility`, `legal`.
- Tauri command `get_prompt(id: String) -> Result<String, String>` — calls `load_prompt`.

**Input validation requirement (operator addition):** `get_prompt(id: String)` is renderer-supplied. The id must be validated against the enumerated list returned by `list_prompts()` — never passed through to the filesystem as a free path. After CSP-bypass XSS, an unvalidated id is a path-traversal sink into bundled resources. The Phase 4 DoD greps the `get_prompt` body for one of `list_prompts | VALID_PROMPT_IDS | prompt_id_allowed | known_ids` (an enumeration-style check) before allowing the function-body anti-stub check to pass.

**Modify `src/components/Workbench.tsx`** — add a "Prompt Library" dropdown in the draft generation area that lists prompts via `list_prompts`. Selecting one pastes the prompt body into the system-prompt field for the next Ollama call.

## 4c. Daily Scan (replaces "OSINT engine")

**Create `src-tauri/migrations/0005_daily_scans.sql`** (UTF-8 LF):

```sql
CREATE TABLE daily_scan_runs (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  run_date TEXT NOT NULL,
  city TEXT NOT NULL,
  state TEXT NOT NULL,
  model_used TEXT NOT NULL,
  prompt_id TEXT NOT NULL,
  raw_response TEXT NOT NULL,
  created_at TEXT NOT NULL DEFAULT (datetime('now'))
);
CREATE TABLE daily_scan_leads (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  run_id INTEGER NOT NULL REFERENCES daily_scan_runs(id) ON DELETE CASCADE,
  rank INTEGER NOT NULL,
  tier TEXT NOT NULL,
  headline TEXT NOT NULL,
  details TEXT NOT NULL,
  source TEXT,
  url TEXT,
  confidence TEXT,
  action TEXT,
  beat TEXT
);
```

**Create `src-tauri/src/core/daily_scan.rs`** with:

- `run_daily_scan(city: String, state: String, since_hours: u32) -> Result<DailyScanRun, String>`:
  1. Pull all `evidence_items` from the last `since_hours` hours (default 24).
  2. Load the aggregator prompt: `load_prompt("aggregator/01-daily-scan")`.
  3. Substitute `[YOUR_CITY]`, `[YOUR_STATE]`, `[YOUR_CITY_AGENDA_PORTAL_URL]` placeholders from the community profile and any source URL with `tier=official_record`.
  4. Concatenate evidence excerpts into a single context block (truncate at ~32 KB to fit context windows of local 9B models).
  5. Call `call_local_ollama` with the model from settings (default `gemma2:9b`).
  6. Parse the response as a structured list of leads. The prompt's output format is well-defined (15-25 leads, each with `Headline:`, `Tier:`, `Source:`, `URL:`, `Confidence:`, `Action:`, `Beat:`, `Details:`). Use a Rust parser that's permissive about whitespace and Markdown decoration; reject only if zero leads are extracted.
  7. Insert one `daily_scan_runs` row + N `daily_scan_leads` rows.
  8. Return the run + leads.

**Input validation requirement (operator addition):**

- `city` and `state` are substituted into prompt placeholders, not used for filesystem / SQL / exec. Still validate as short ASCII (regex `^[A-Za-z][A-Za-z .'-]{0,63}$` or equivalent). Reject otherwise.
- `since_hours` MUST be bounded `0 < since_hours <= 168` (one week) to prevent unbounded evidence pulls.

**Tauri command** `run_daily_scan` exposed to the frontend.

**Modify `src/components/SystemStatus.tsx`** — add a "Run Daily Scan" button. Disabled if Ollama unreachable. On click, calls `run_daily_scan`, navigates to a new `DailyScanResults` view.

**Create `src/components/DailyScanResults.tsx`** — displays the leads in rank order with tier badges. Each lead has an "Open in Workbench" action that creates a draft `lead` row from the scan lead (links back via a `from_scan_lead_id` column on `leads` — add via the same migration).

## 4d. Plain Language Rewrite

**Tauri command** `plain_language_rewrite(draft_id: i32) -> Result<String, String>`:

1. Load the draft.
2. Load `prompts/story/07-plain-language.md`.
3. Call Ollama with the prompt as system and the draft as user input.
4. Return the rewritten text. Do NOT persist automatically — the UI decides whether to save it as a new draft variant.

**Modify `src/components/Workbench.tsx`** — add a "Rewrite for Newsletter / Email / Facebook" button on every saved draft. Calls `plain_language_rewrite`, displays output in a side panel, offers "Save as variant" or "Copy to clipboard".
