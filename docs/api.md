# Loopback HTTP API Reference

CivicNewspaper's Rust backend runs an [Axum](https://github.com/tokio-rs/axum) HTTP
server bound **strictly to the loopback interface** `127.0.0.1:12053`
(`src-tauri/src/core/server.rs`). It exists so local clients — the Chromium
browser extension and IDE-side assistant skills (`assistant-skill/client.js`) —
can pair with the desktop app and exchange data over a controlled, authenticated
channel.

This file is the route contract. It is generated by hand from `server.rs`
(routes + JSON shapes) and `auth.rs` (the middleware rules). If those files
change, update this doc.

> **Trust boundary:** the user account, not the process. Any process running as
> the same OS user can reach `127.0.0.1:12053`. The bearer token is the only
> gate. See [SECURITY.md](../SECURITY.md) "Known weak spots" for the full threat
> model.

## Base URL

```
http://127.0.0.1:12053
```

The server binds `127.0.0.1` only — it is never reachable from another machine.

## Request prerequisites (enforced by `auth_middleware`)

Every request except `POST /api/pair` passes through `auth_middleware`
(`src-tauri/src/core/auth.rs`). It applies three gates, in order:

1. **Host header** — must be exactly `127.0.0.1:12053`. Any other value
   (including a DNS name that resolves to loopback) returns `403 Forbidden`.
   This defeats DNS-rebinding attacks.
2. **Origin header** — if an `Origin` header is present, it must start with
   `chrome-extension://`. A literal `Origin: null` is rejected. A request with
   **no** `Origin` header is allowed (this is how IDE/CLI agents, which send no
   Origin, authenticate). A present-but-untrusted Origin returns `403 Forbidden`.
   - Exception: `POST /api/pair` skips the Origin check when the request carries
     the `x-civicnews-pair` header (so a non-browser client can pair).
3. **Bearer token** — every route under `/api/*` except `/api/pair` requires
   `Authorization: Bearer <token>`, where `<token>` is the long-lived token
   issued by `/api/pair`. A missing/malformed header returns `401 Unauthorized`;
   a token not found in (or revoked from) the `paired_clients` table returns
   `401 Unauthorized`.

`POST /api/pair` is the only unauthenticated route: it has no bearer-token
requirement (that is the route that *issues* tokens) and is additionally
IP-rate-limited (see below).

## Routes

| Method | Path | Auth | Purpose |
|---|---|---|---|
| POST | `/api/pair` | none (rate-limited) | Exchange the one-time pairing token for a long-lived bearer token |
| GET | `/api/queue` | Bearer | List all leads and drafts |
| GET | `/api/evidence/:lead_id` | Bearer | List evidence items behind a lead |
| POST | `/api/drafts` | Bearer | Create a new draft (forced to `draft_generated` status) |
| POST | `/api/llm/task` | Bearer | Run a one-shot local-LLM prompt |
| POST | `/api/guardrails/check` | Bearer | Run the guardrail lint over a draft |

---

### `POST /api/pair`

Exchanges the one-time pairing token (the 22-character URL-safe base64 string
shown in the desktop app's **Browser Pairing** tab) for a long-lived bearer
token. No `Authorization` header is required. The desktop app generates the
pairing token by taking 16 bytes from `OsRng`, base64-URL-safe-no-pad encoding
them (22 chars), and storing the SHA-256 hash; this route hashes the submitted
value and matches it.

**Rate limit:** 5 failed attempts per client IP per 1800 seconds (30 min).
Exceeding the limit returns `429 Too Many Requests`. A successful pair resets the
counter for that IP.

**Request body**

```json
{ "pin": "<22-char pairing token>" }
```

(The field is named `pin` for historical reasons; the value is the 22-char
pairing token, not a numeric PIN.)

**Response `200 OK`**

```json
{ "token": "<long-lived bearer token>" }
```

**Errors:** `401 Unauthorized` (token does not match), `429 Too Many Requests`
(rate limit), `500 Internal Server Error`.

---

### `GET /api/queue`

Returns every lead and every draft in the local database.

**Response `200 OK`**

```json
{
  "leads": [
    {
      "id": 1,
      "detector_name": "Money Threshold",
      "why": "…",
      "confidence": "medium",
      "risk_level": "low",
      "confirmation_checklist": "[]",
      "from_scan_lead_id": null,
      "created_at": "2026-05-28T12:00:00Z"
    }
  ],
  "drafts": [
    {
      "id": 1,
      "lead_id": 1,
      "format": "brief",
      "title": "…",
      "content": "…",
      "status": "draft_generated",
      "verification_checklist": "[]",
      "missing_evidence_notes": null,
      "correction_note": null,
      "created_at": "2026-05-28T12:00:00Z",
      "updated_at": "2026-05-28T12:00:00Z"
    }
  ]
}
```

`confirmation_checklist` and `verification_checklist` are JSON strings (a
serialized array), not nested JSON.

---

### `GET /api/evidence/:lead_id`

Returns the evidence items associated with the given lead id (path parameter,
integer).

**Response `200 OK`**

```json
[
  {
    "id": 1,
    "source_id": 3,
    "url": "https://example.gov/minutes/2026-05",
    "fetched_at": "2026-05-28T12:00:00Z",
    "excerpt": "…",
    "content_hash": "b94d27b9…",
    "entities": "[\"City Council\"]"
  }
]
```

`entities` is a JSON string (a serialized array of strings).

---

### `POST /api/drafts`

Creates a new draft. The server forces `status` to `draft_generated` regardless
of input — a client cannot create a draft in an approved/published state.

**Request body**

```json
{
  "lead_id": 1,
  "format": "brief",
  "title": "…",
  "content": "…",
  "verification_checklist": "[]"
}
```

`lead_id` and `verification_checklist` are optional. If
`verification_checklist` is omitted it defaults to `"[]"`.

**Response `200 OK`**

```json
{ "id": 42 }
```

**Errors:** `500 Internal Server Error`.

---

### `POST /api/llm/task`

Runs a single prompt against the local Ollama model selected in Settings
(falling back to the default model if none is selected). This is a thin pass
through to the local LLM — there is no streaming over this route.

**Request body**

```json
{
  "prompt": "…",
  "system": "…"
}
```

Both fields are required.

**Response `200 OK`**

```json
{ "result": "<model output>" }
```

**Errors:** `503 Service Unavailable` (the local Ollama instance could not be
reached or the call failed).

---

### `POST /api/guardrails/check`

Runs the guardrail lint over an existing draft and returns the findings. These
are advisory warnings only — the guardrails never block compilation or a status
change.

**Request body**

```json
{ "draft_id": 42 }
```

**Response `200 OK`**

```json
{
  "is_clean": false,
  "issues": [
    {
      "category": "Verbatim Overlap",
      "message": "…",
      "severity": "warning",
      "paragraph_index": 3
    }
  ]
}
```

`category` is one of `Citation Coverage`, `Accusatory Language`,
`Verbatim Overlap`, `Legal Naming`. `severity` is `warning` or `error`.
`is_clean` is `true` when `issues` is empty.

**Errors:** `500 Internal Server Error` (e.g. the draft id does not exist).

## Status codes summary

| Code | Meaning in this API |
|---|---|
| `200 OK` | Success |
| `401 Unauthorized` | Missing/malformed/invalid bearer token, or wrong pairing token on `/api/pair` |
| `403 Forbidden` | Bad `Host` header, or untrusted `Origin` header |
| `429 Too Many Requests` | `/api/pair` rate limit hit (5 fails / 30 min / IP) |
| `503 Service Unavailable` | `/api/llm/task` could not reach the local Ollama instance |
| `500 Internal Server Error` | Database or internal failure |
