## Required Retest Scope

Run a full cleanroom end-to-end test, not a narrow smoke test.

1. Verify the public landing page at:

   `https://scottconverse.github.io/CivicNewspaper/`

   Required checks:

   - here.now is visible in the first-page product/publishing messaging
   - the page does not contain `Publish Statically`
   - the page does not mention Vercel as a recommended drag-and-drop default
   - platform download copy is conditional and does not falsely claim all platform installers exist for every release
   - page has no obvious layout trap on desktop or mobile
   - capture desktop and mobile screenshots

2. Verify installer identity:

   - installer hash matches the value above
   - app launches as The Civic Desk
   - app version and installed artifact are recorded

3. Run first-use setup as a normal user:

   - configure Longmont, Colorado
   - allow the app to inspect hardware and choose an appropriate local AI setup
   - if the app needs Ollama/model setup, the app must drive it and show progress
   - do not install Ollama or models manually outside the app

4. Source intake:

   - include official Longmont/county/state civic sources
   - include public, no-login social/community sources such as public Reddit, public YouTube/meeting pages/transcripts when available, public local media, and public forums
   - import available source-list fixtures if present or generated by the app
   - confirm reviewable source candidates and source types are visible

5. Daily Scan and lead quality:

   - run Daily Scan from the UI
   - require deterministic source fetch/evidence capture before AI summarization
   - expect 10-25 useful leads when source volume allows
   - duplicate/paraphrased leads should be clearly reduced or merged
   - evergreen background pages must not be treated as finished stories without a current development
   - capture the full lead list and any warnings

6. Workbench and editorial workflow:

   - generate drafts from multiple leads using the app's configured local AI
   - verify draft titles are real headlines, not long lead summary sentences
   - verify public story bodies do not expose `Headline:`, `Nut graf:`, `Reporting Steps`, `[Source needed]`, `[Verification needed]`, or `[End of Report]` as reader-facing article copy
   - run guardrails
   - run the press-freedom/legal-risk advisor on at least one story
   - exercise approve, hold, send back / needs more work, and return-to-queue paths
   - confirm no software veto prevents the editor from making the final decision

7. Publication output:

   - produce a complete issue with 10-25 leads and 5-10 reader-facing stories/briefs if source material supports it
   - if fewer are possible, report exactly which stage constrained the output and why
   - compile the static site
   - export ZIP
   - publish anonymously to here.now
   - save the here.now URL
   - save the local output folder path and ZIP path
   - capture homepage, article page, RSS/feed, share package, mobile layout, and print layout evidence

8. Output quality audit:

   - no duplicate story topics in the public issue unless intentionally labeled as updates/related items
   - no reporter-note scaffolding in public pages
   - no generic evergreen rewrite sold as current news
   - headlines should read like headlines
   - every article/brief should clearly show why it matters now, source basis, and what remains unverified when applicable
   - source/evidence display should be readable

9. Mojibake/encoding audit:

   - scan visible UI, generated text artifacts, static HTML, RSS, Markdown share files, and report files for mojibake patterns
   - include `LONGMONT / CO` or equivalent city/state display evidence
   - report any non-ASCII that is intentional vs broken encoding

10. Final report:
